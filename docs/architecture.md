# Architecture

## 1. Architecture Principle

The MVP is static-first.

The app does not need a backend to prove the core product. Cards, tasks, topics, and validation can live in the repository. User progress can start in local state or `localStorage` and later move to a backend without changing the content schema.

## 2. Recommended Stack

- Vite.
- React.
- TypeScript.
- CSS Modules or plain CSS.
- Static JSON data generated by import scripts.
- `localStorage` for optional local preferences/progress.

Avoid for MVP:

- Backend framework.
- Database.
- Authentication.
- Global state libraries unless the app grows beyond simple local state.
- Heavy UI frameworks.

## 3. Current Content Pipeline

JavaScript interview material:

```txt
external/sudheerj-javascript-interview-questions
  -> tools/import-sudheerj.py
  -> data/cards/*.json
  -> data/coding-exercise/sudheerj/**
```

Data structures and algorithms:

```txt
external/sudheerj-datastructures-algorithms
  -> tools/import-sudheerj-dsa.py
  -> data/dsa/*.json
  -> data/coding-exercise/sudheerj-dsa/**
```

Import scripts are the only place that should reshape external source repositories.

## 5. State Model

Training session state:

```ts
type TrainingMode = "random" | "topic";

type TrainingSession = {
  mode: TrainingMode;
  topicId?: TopicId;
  cardIds: string[];
  currentIndex: number;
  flippedByCardId: Record<string, boolean>;
};
```

Task session state for later coding-practice mode:

```ts
type TaskSession = {
  dataset?: "javascript" | "dsa";
  topicId?: TopicId;
  taskType?: TaskType;
  taskIds: string[];
  currentIndex: number;
  isSolutionVisible: boolean;
};
```

## 6. Session Behavior

Starting a session:

1. Select source content.
2. Filter by topic if needed.
3. Exclude cards marked `known`.
4. Shuffle for random mode, or keep topic order if an ordered mode is added later.
5. Store IDs in session state.
6. Set `currentIndex` to `0`.

Moving to next card:

1. Increment `currentIndex`.
2. Wrap around when moving beyond the end.
3. Preserve per-card flip state.

Going back:

1. Decrement `currentIndex`.
2. Wrap around when moving before the first card.
3. Preserve per-card flip state.

Shuffle:

1. Shuffle the current filtered source.
2. Reset index to `0`.
3. Hide answer.

## 7. Content Loading

Load generated JSON explicitly. The app should treat imported content as read-only static data.

```ts
import jsIndex from "../../data/cards/index.json";
import dsaIndex from "../../data/dsa/index.json";

export const indexes = [jsIndex, dsaIndex];
```

The exact import style can change depending on Vite configuration, but the boundary should stay simple:

- Generated JSON is content.
- React components render sessions.
- User progress is separate local state.
- Import scripts reshape external source repositories.

When source repos are updated, rerun the corresponding importer and validate JSON.

## 8. Validation

Validation must remain part of the content flow.

Validate:

- Unique card IDs.
- Unique task IDs.
- Unique card questions.
- Required fields.
- Topic IDs.
- Source metadata.
- JSON parseability.

Current manual validation command:

```sh
jq empty data/cards/*.json data/dsa/*.json
```

## 9. UI Layout

The first screen should be the product itself.

Recommended layout:

- Top compact action row with theme toggle, language toggle, topic buttons, and Random.
- Main card area.
- Bottom or side controls depending on viewport.
- No product title, top nav tabs, dashboard, normal-card modal, or drawer.

Desktop:

- The card slider is vertically centered.
- The previous and next cards are visible beside the active card.
- The active card is the visual focus and roughly 1.5x larger than side cards.
- Card width aligns with the compact top row rhythm and remains readable.

Mobile:

- Side cards are hidden.
- The active card uses the free viewport height.
- Bottom controls stay accessible.
- Horizontal swipe changes cards.
- Code snippets scroll horizontally if needed.

## 10. Component Responsibilities

`TrainerPage`

- Owns trainer screen composition.
- Connects controls to session hook.

`useTrainingSession`

- Builds session card list.
- Handles reveal, next, previous, restart, shuffle.
- Does not render UI.

`Flashcard`

- Renders front/back card state.
- Receives data and visibility as props.
- Does not know how sessions are created.
- Renders typed content blocks explicitly; does not guess code/table/list from raw strings.

`TasksPage`

- Same role as `TrainerPage`, but for practical tasks.

## 11. Future Backend Boundary

When backend is added, keep static content and user progress separate.

Backend should own:

- User identity.
- Review history.
- Card progress.
- Bookmarks.
- Custom decks if needed.

Frontend/repository can still own:

- Canonical built-in card content.
- Topic metadata.
- Static task definitions.

Do not store user progress inside card objects.

## 12. Account And Feedback Backend

The first backend-backed layer should add user identity, reports, profile, and synced progress without changing static content loading.

Frontend still owns:

- Rendering cards.
- Static card indexes.
- Topic filters.
- Local guest mode.

Backend owns:

- Authenticated user profile.
- User role: `user` or `admin`.
- Synced card progress.
- Error reports.
- Active report limit.
- Admin report status changes.

Recommended API boundary:

```txt
GET    /api/me
PATCH  /api/me

GET    /api/me/progress
PATCH  /api/me/progress/:cardId
DELETE /api/me/progress

GET    /api/me/reports
POST   /api/reports

GET    /api/admin/reports
PATCH  /api/admin/reports/:reportId
```

Important constraints:

- Guests can keep local progress, but persistent reports require sign-in.
- A user can have at most 2 active reports where status is `open` or `in_review`.
- Only admins can see all reports.
- Users can see only their own reports.
- Resetting progress deletes only user progress, not reports, profile, or static cards.

Implementation should start with UI and local mock state, then move persistence behind the API boundary.
